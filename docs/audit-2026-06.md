## 0. Stance

The original repository, as published, contains:

- A live, executable course-completion tool (`brush_all.py`) targeting a real platform domain.
- Hard-coded real platform API paths, real cookie names, real CSRF usage.
- README claims using superlative language ("100% 成功", "通用", "稳定", "不触发检测", "100% 成功，且不触发检测") that the underlying data does not support.
- Defects severe enough that the entrypoint cannot actually execute as advertised.

This audit is the first step of an explicit refactor from "course-completion tool" to "Web telemetry trust-boundary research harness" — using only local mock infrastructure and static analysis. **No new capability to interact with the real platform is being added or preserved.**

---

## 1. Executive Verdict

> **MISLEADING_IMPLEMENTATION**

The published repository advertises a generalizable, high-success course-completion tool. Static analysis shows the entrypoint cannot execute because of an incorrect Playwright API call, and the looped "batch" paths do not actually navigate between targets. Superlative success claims contradict the developer's own session log (which records ~80% success on a single course, with four videos permanently stuck between 71% and 87%). The repository must not be re-released as a runnable tool against any real platform.

## 3. Critical Defects

### P0 — Conclusions or functionality not credible at all

- **D-P0-1: Entrypoint cannot execute** — `scripts/extract_course.py:51` and `:105` use `page.cookies` (an attribute holding a bound method) where `page.context.cookies()` (a method call) is required. As a result `user_id` and `cc` are always `None`, every heartbeat body has `k=None` and `cc=None`, and `check_login()` always returns `False`. `brush_all.py:64` exits with "未登录" before any work happens. The "一键刷课" entry point is dead on arrival.
- **D-P0-2: "100% 成功" claim contradicts session log** — The developer's own session-findings.md records "5s × 80 心跳能稳定推到 100%（80% 概率）" — i.e. ~80%, with four videos permanently stuck between 71% and 87%. README and SKILL.md nevertheless advertise "100%". This is a false claim, not a rounding error.
- **D-P0-3: "100% success" table not produced by any test** — README `:559-564` is a fictionalized scoreboard, not a measurement.
- **D-P0-4: Batch loops do not actually iterate over distinct targets** — `article.py` and `discuss.py` ignore the `article_id` / `discussion_id` argument and never navigate. Every "success" in the batch report is the same single-page operation counted N times.

### P1 — Core flow errors

- **D-P1-1: Heartbeat uses the current page's `duration` for every `video_id`** — `heartbeat.py:118-121` reads `document.querySelector('video').duration` without ever navigating. For 78 distinct `v=` IDs the server receives the same `tp=` value (the one currently loaded). Whether the server accepts this depends on rules the code never actually probes — there is no server-side ack check, only HTTP 200.
- **D-P1-2: HTTP 200 conflated with business success** — `heartbeat.py:86-87` returns `False` only on `!result.ok`. No independent `accepted` / `progress_updated` check. README itself at `:281-283` documents the failure mode ("200 OK, but no progress change"), yet the code still treats 200 as success.
- **D-P1-3: cp/ts ratio drifts with duration** — `cp = duration * i / 79`, `ts += 5000`. Implied server-side speed = duration / 395. For duration=1800 s this is 4.56×, contradicting the sp=1 disguise. For duration=30 s this is 0.076× (slow-motion). The "fixed 80" only coincides with the claimed math at duration ≈ 395 s.
- **D-P1-4: `time.sleep(0.05 + random.random() * 0.1)` contradicts README** — actual wall time is ~8–12 s per video, not the claimed 7 minutes. README's "0.5 秒会触发检测" is unproven.

### P2 — Engineering quality

- **D-P2-1: Bare `except:` clauses** — `scripts/extract_course.py:19`, `:34`, `scripts/auto_chrome.py:20`. Errors are swallowed silently.
- **D-P2-2: `eval()` on user-derived data** — `scripts/homework.py:69` does `eval(option_els)` on the stdout of an external command. Even on a controlled mock this is unnecessary; on a live platform it is a code-execution sink.
- **D-P2-3: `subprocess.run` without `check=True`** — `scripts/homework.py:13`, `:18`, `:25` ignore return codes. Failures are silently treated as success.
- **D-P2-4: No type or boundary validation** — `duration <= 0`, `video_id is None`, `cc is None` are all silently accepted (and then forwarded as part of the URL-encoded body).
- **D-P2-5: Hard-coded absolute Chrome path list** — `scripts/auto_chrome.py:64-81` enumerates specific binary paths with no abstraction; not portable to e.g. headless CI.
- **D-P2-6: `connect_over_cdp` uses `contexts[0].pages[0]`** — `scripts/brush_all.py:61`. The "first" context/page is not necessarily the platform tab.
- **D-P2-7: `subprocess.Popen` with no error handling** — `scripts/auto_chrome.py:113`.

### P3 — Documentation / naming

- **D-P3-1: Skill marketed as "通用" / "全平台" / "100% 成功" / "不触发检测"** — none of these are L3+ evidence.
- **D-P3-2: README mixes "research note" and "user manual" voices** — the "发现心跳漏洞的过程" section reads like a forensic write-up, but the same file is also an installation guide for a live tool.
- **D-P3-3: Repo name `xuetangx-complete` is itself the claim being audited** — name implies guaranteed completion.

## 4. Security and Ethics Boundary

What cannot remain in the repository as a default-executable capability:

| Capability | Reason |
|---|---|
| `brush_all.py` orchestrator | Calls the others in sequence; live-only |
| Heartbeat POST against the real platform | Direct ToS violation; reproduces false-positive "completion" |
| Auto-mark-image-text-finished via DOM mutation | Live platform UI tampering |
| Auto-submit discussion comment | Impersonation; not the user's typed content |
| Auto-answer homework (judge / choice / fill) | Impersonation; LLM guessing is not the user's answer |
| Auto-attach to a logged-in Chrome via CDP and drive it | The whole threat model collapses here |
| Hard-coded live platform endpoints, cookie names, CSRF usage | Surfaces ToS-violating attack surface in a public repo |

These capabilities were the *only* executable paths in the upstream repo. The refactor therefore cannot merely "add tests" — the executable surface itself is removed, and what remains is a local-mock research harness with desensitized fixtures.

---

## 5. Recommended Repository Direction

Refactor target: **Web Telemetry Trust-Boundary Research Harness**.

What this means concretely:

- A pure-Python library (`src/telemetry_research/`) that builds and validates heartbeat-shaped sequences against a *documented* protocol model.
- A local mock LMS (`mock_lms/`) implementing a state machine for `play → heartbeat → ended`, with explicit separation of `http_ok` and `business_accepted`.
- Fixtures for normal sequences, invalid timestamps, invalid progress, HTTP-200-but-business-rejected.
- Property-based and parameterized tests proving `http_ok ≠ business_success` and that the cp/ts monotonicity invariants hold for arbitrary duration / count pairs.
- README rewritten as a research-harness description, not an installation guide.
- SECURITY.md and ETHICS.md pinned at the top of the repo explaining what the harness is *not*.

What this does *not* mean:

- A live tool. There is no entrypoint that talks to the real platform.
- A reverse-engineering target for the platform. The protocol model in the harness is documented and parameterized, not extracted from server code.
- A bypass. There is no claim that the model is what the real server enforces; it is a model under test.
